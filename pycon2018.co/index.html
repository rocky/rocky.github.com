Programmer vs. Computer

A programmer may write: 


  x + y

In the C Python implementation, the code you write gets translated into something called Bytecode. when I write in Python x+y, next the computer sees... next

The above is for C Python implementation, and the instructions are called Bytecode. The above bytecode is for Python 3.6.

The "2" on the first line means that the following instructions come from line two, until there is another line number given. But since there are no other line numbers given, the Python interpreter assumes that all of the three instructions have something to do with line 2.

The numbers 8, 10, and 12 are the instruction addresses, or really instruction offsets from where the function or module starts. So being more precise about what is associated with line 2, the answer is instructions at offsets 8, 10, and 12. In Python versions before 3.6, the instruction size for `LOAD_NAME` is 3 bytes instead of 2, so the offsets would be 8, 11, and 14.

And this (or a more obstruse binary-encoded version of that) is what the computer "sees" and runs.

In this simple example, it looks like a single token in the source-code gets converted to a single line in the bytecode. However the order is a little bit different: the "add" comes at the end rather than in the middle.

but the computer sees:

Side-by-side Translation

1: Whose woods these are I think I know.
2: His house is in the village though

1: A quién pertenece este bosque creo que sé
2: Aunque su casa en el pueblo está

You can assume that the first line of the English is roughly like the first line of the Spanish, and the second line of the English is like the second line of the Spanish. And possibly the words at the beginning or end of the sentence roughly go with words at the beginning of the corresponding translation. "His house" at the beginning of line two is almost the same position as "su casa." But not exactly, and sometimes the position doesn't match at all. For example, see the positions of "is" versus "está" on line two. Also "though" versus "Aunque".

And this is sort of like what goes on when you get the error message that we saw above when it said the error was on line 2.

So let's do a side-by-side translation in Python...

Side-by-side Python translation


  x = 1  # line 1
  y = 2  # line 2
		

  1: 0 LOAD_CONST  1
     2 STORE_NAME  x

  2: 4 LOAD_CONST  2
     6 STORE_NAME  y

Notice again that the order of the variable name and constants have been reversed as they were in the example with addition. The mathematical name for this kind of ordering is called "reverse polish notation". English adjectives are like that.

Now that we have seen how this resembles a human-language translation problem, rather than a programmer-versus-computer problem, we can ask: how would we get more detail in the correspondences between the two translations?

The tree structure is roughly the same, but rename "Name" and "Load" are both called "Expr" in the reconstructed tree. The name change isn't important. The main thing is that in either case they alway have only one child, and they can derived from and also derive a "sucbscript".

Notice how all of the leaves of the parse tree are the bytecode instructions. It might be hard to see that because the leaves are at different tree depths. But as you will see, the different depths will become important a little later.

And as we saw with "Entiendo" versus "I understand", there is an extra instruction: "SUBSCR".

Parse to Text

Here we'll show how to reconstruct the source text from this tree.

Recall the intructions are the leaves. they are the things that have offset numbers, 6, 8, ... In addition to that offset value, each Python bytecode instruction must have an operand value for some instructions. And for other instructions there will never be a value. Both "NAME" and "CONST" have an operand value. In our example both "NAME" at offset 6 and 8 both have the value "prev" for its operand. The "CONST" has a value 0 in our example. In contrast, "SUBSCR" never has an operand.

One other thing to note here is that there only two different node types: Expr and Subscript. down

In this tree, I've take the operand values of the instruction, when there is one, and shown that instead of the instruction name and offset.

For the other nodes, recall they are either Expr or Subscript. For Expr, just copy the value from its child below; for Subscript, take the value from the first child and the surround the value of the second child in square brackets and finally ignore the last child which here is always "SUBSCR". Debugging code when there is no no source

The last example I am going to show is debugging bytecode when there is no source code at all! Just the bytecode file. Reverse engineers: pay attention.

There is a command-line program I wrote that will disassemble bytecode. It has a --header option to give meta information about a bytecode file. In particular, the version interpreter version and the magic number of the for the bytecode file. Alos it shows the file that was used to create it. Filename "exec" is pretty meaningless.

Even though I don't have the source code, I can still run the program Python.

But what if I try to debug it using the pdb module, I get gibberish. Python thinks the bytecode is text. I can't even issue a quit command or Ctrl-d to get out. I have to suspend and kill it.

I've been running this inside Emacs, and I also I'm going to show an emacs package I wrote which interfaces not just to this debugger, trepan3k, but also to about a dozen others including pdb, and ipdb. The package is called realgud.

The emacs package tracks the location for me. The green orange and brown arrows in the fringe indicate where the program is where has been. The realgud track all the places you've stopped.

Notice that in function fib() there are two recursive calls. And again deparse can show you which one was called.

And as you might expect in a debugger front-end, you highlight a region and have that evaluated.
