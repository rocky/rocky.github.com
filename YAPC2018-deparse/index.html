Programmer to Computer

A programmer may write: 


$a / $b / $c

In order to solve this problem, we need to go into a little of the nitty-gritty of how Perl5 works. If you are familiar with compilers, this will look familiar.

In the Perl5 implementation, the code you write gets translated some thing called a B::OP tree, and running a Perl program then is simply walking the tree and performing various actions along the way. In short we say we "evaluate" the tree. If there are loops in the tree, some of the tree nodes may get executed repeatedly for each iteration of a loop.

So let's see the B::OP tree for this...next

There are a couple of ways B::Consise can display an OP::Tree, I have chosen the tree-oriented way rather than a linear-sequence. So as a result you'll see numbers at the left which indicate the instruction sequence were it shown in the linear ordering. And on the extreme right, the number which is one more than the number on the extreme left, if that exists.

So note that the first instruction listed is in fact the last one in the sequence. Interesting. And then there is a skip to from the last instruction to the first one; then the second instruction and a nother skip to an instruction near the end. Those of you who are astute might see the pattern here. We'll come back to this later.

I have simplified the instructions a little by removing instructions that have been replace by some other instruction or "optimized" away. But even with removing instructions, you'll see that the B::OP Tree

Instruction 2, the "nextstate" instruction, isn't an instruction at all. It simply marks a statement boundary, and contains values for a file name and line number.

When you are tracing instructions such as inside a debugger, that instruction is called "dbstate" instead. In fact that's how Perl knows whether to stop or not.

All the instructions, in instruction order, after a nextstate or dbstate up to the next nextstate/dbstate are associated earlier nextstate/dbstate instruction. In particular, all of the instructions from instruction 2 onto 8 are associated with. the single nextstate listed. Specifically they are all considered to be "on line 1" of "div.pl".

And now you should see what the problem is. There are two "divide" B::Ops in that sequence of instructions, and what we want to do now is say a little more about each other than they come from the same line.

Since Gaelic and English are so close, for large stretches you can do a word-for-word translation for the most part.

In this particular example words at the begining of one sentence match words at the beginning of the other, and those at the end of Gaelic match those at the end of English.

And I've used this hueristic in debugging my Perl and C programs where I find guess that if the address is near the preceding nextstate, it is probably towards the beginning of the source code.

Athough it is pretty much the same thing as with the side-by-side translation, there is one one caveat...

Let's ignore the additional "nextstate" instruction which is more like a meta-instruction than an instruction that does a computation.

Instead of the order "variable", "equals" (or "assign"), "constant", we have in instructions "constant", "variable", "assign". And in general the operator "assign" comes last. So in contrast the the Gaelic English translation, the first instruction of the Optree translation is not match first token of the Perl code nor can the same be said of the last token.

Ths kind of operator-last arrangement is called Polish-Prefix notation, used also HP calculators of the days of your, or the programming languages Forth, or Postscript. A recent language that uses Polish Prefix is the Bitcoin Script language.

The advantange of listing instructions this way is simple: by the time you get to the operation to be peformed, assignment, all of the data is available on an execution stack.

Now that we have explored how this resembles a human-language translation problem, rather than a programmer-versus-computer problem, we can ask: how would we get more detail in the correspondences between the two translations?

In human language translation you would diagram a grammar on top of the sentence and then match grammary constructs between the two translations.

Perl already provides this tree structure which pretty much follows its source-code grammar.

Parse to Text

Here is a diagram of the tree we saw before. It is not an "abstract" syntax tree, but it is a syntax tree.

Now we'll show how to reconstruct the source text from this tree. The is in fact how the Perl module B::Deparse works. B::Deparse has been available in Perl for 20 years or so. B::DeparseTree is my extension to B::Deparse which saves additional information so as to be able to pick out the fragments for a given B::Op instruction.

For those familiar with compilers, this sort of thing should look familiar. Using the node type and information propagated through adjacent nodes, we can create a Perl source-code string that characterizes what the tree rooted at that node does.

I have written the instruction order near each node in the tree. However we can think of that as a memory address as well. And then the game becomes, if I give you an instruction number or memory address what Perl source-code fragment does that represent?

Here we have only two node types: "gvsv" and "divide". In B::Deparse the information obtained a node's children are strings of Perl source-code fragments. Information passed down from above is an operator-precedence number that is used to determine whether to surround the final result in parenthesis.

In B::DeparseTree we need to keep more information. Keep a hash of node addresses, so you can query information about any node in the tree. We also pass down the parent's address so that we can walk up the tree. This is useful so that we can give context around the point of interest.

Athough this may not be needed in the future, right now, instead of passing back strings, to a parent, we pass back our own kind of tree node structaure which has richer information about the text fragment. This help not only in debugging, but also it allows us to make more sophisticaed decisions about how to format the code. For example if the fragment is a block table with a large number of entries that may change how we might want to format the list parameter after the block in a grep expression.

In this diagram, I've given a Perl source-code translation for each node.

The way I got this was to applied to rules for gvsv and divide to each of the nodes. I have kept the instruction numbers from the last slide, to show that how that given an instruction number, we can report what Perl code is associated with that.

So if I am at instruction 3, the translation to Perl would be "$a"; if at instruction 4, the "$b". At instruction 5 we get "$a/$b". At instruction 7 we have "$a/$b/$c". And with this you now can see how a program can tell you which divide you were working on.

As for the rules or Perl code which I used to create these strings, I'll give that code next, but it doesn't take too much imagination to see what's going on: for "gvsv" you pick out the name of the variable which and that has dollar at the beginning. For "divide" you take the string for the left child and right child but slap a "/" in between them.

I give some specific Perl code next, but before leaving this slide, let me just point out that the string for each node is created purely on information from the node's parent or children. In compiler terminology this is called "inherited" or "synthesized" attributes. down

Perl fragments associated with B::OPs

divide rule


sub divide {
    my ($self, $op, $context) = @_;
    my $left = $self->deparse($op->first);
    my $right = $self->deparse($op->last);

    my $str =  $left . '/' . $right;
    return $self->maybe_parens($str, $context, 19)
}

This code is a simplfication of what B::Deparse does. There isn't a rule for "divide" specifically, but instead the class B::OP's that is part of. It is a binary operator or "BINOP".

The are other complications as well which I am not showing. The above gives you a sense of how this works. The thing I want to stress in the code above that resemmbles the B::Deparse code, is that a string is built up over the course of executing the subroutine. Also also there is a fair bit of boilerplate code in getting strings from the child nodes

The line at the bottom is interesting in that it makes use of information from the parent to decide whether or not to enclose the string in parenthesis. If this were and addition operator, but my parent was a multiplication operator, its context value would be higher than mine, 19, so I'd know that parenthesis is need around the addition operation. next

For gvsv OPs, all we do extract the name. Again this is a simplification. next

sub gvsv {

   my $str = $self->gv_name($op);

Simplifying the 100's of rule codes

B::Deparse.pm is about 5K lines of code excluding the POD documentation at the bottom. A fair bit of it is like the code I showed above. How can we reduce the above?

Some of you might have already thought that we might use sprintf instead of strict concatenation. I like the sprintf because it emphasizes the idea more that a good organization of a rule is to gather data first, and then use the data at the end to format it.

So the way to reduce that long repetitive code, is to create a little Domain-Specific Language or DSL

In B::Deparsetree the returned value is a tree, so we can't use sprintf. So instead I worte a little template parser like sprintf, and use "%c" to indicate the value returned from the child node. And when there is precedence involved, I use "%p" which informs the child as it recurses what my precidence is. Here it is 19.

my $str =  sprintf("%s/%s", $left , $right);

Lingering Problem: PC location in a call

The last problem I'll mention has to do with the location of the instruction pointer or "program counter" of the parent of a call. The instruction pointer is advanced before the call is made. So inside the call if you look up the call stack you'll see the next instruction to be run rather than the call that was made.

This is an example of a bug that has been in Perl for 16 years. One of the cool things about B::Deparsetree is that I can more easily fix this outside of fixing Perl.


foreach( my $x =0 ; $x < 1; $x++) {   # Line 1
  print_line();                       # Line 2
}
sub print_line {
  print "call from line " . [caller(0)]->[2] . "\n";
}

prints "call from line 1".
