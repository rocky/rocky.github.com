One of the interesting things about this photo of the "lab" you see here, taken at dawn, is that you don't see a single person! Anywhere!

The project I was hired for was to help develop a compiler for a newly-designed, fast-running CPU, called 801. This was IBM's — and possibly the world's — first reduced-instruction-set or "RISC" CPU. The research idea was reconsidering or re-thinking the division of labor between what should be done, or rather, what is better done, in a compiler versus what is better done in hardware. In contrast, the Intel x86 instructions, are more "complex". What I mean by this, is that an x86 instruction, like mov which we will see later, is in fact level several "micro" or smaller instructions when run on the hardware chip. There is a layer of "microcode" in the between the machine code input and the chip hardware. This makes it possible for assembly-language and machine code instructions to look the same, independent of the level of sophistication of the chip underneath.

But in a RISC CPU and in the 801 CPU in particular, there was no microcode layer. The low-level guts were exposed and available for the compiler to use, in strange and weird ways. In 1982, In contrast to earlier times, it was felt 1982 that most people did not write in assembly code anymore, but rather wrote in high-level languages using a compiler to bridge the language gap.

At any rate, a key for the success of this RISC CPU was having an "optimizing" compiler, or a compiler that produced fast-running code by taking advantage of this kind of lower, microcode-level, instruction set.

By the way, when the project became successful, the IBM Marketing guys took over. They renamed the 801 CPU chip to [FIST] POW-ER. "801" was simply the building number that you see in this photo.

The financial programmers who wrote code in this environment weren't very good. Their interests were more focused on improving salary than improving program quality. So part of the system admnistrator's job was to prove that the code that we were tasked to run was leaking memory, or I/O resources, or just plain buggy. Invariably, this was the reason our computers were crashing. And if we could not be demonstrate a problem in the program, we'd have to get bigger, costlier, and faster computers with more memory ... And be able to reboot boxes faster...

It is in this environment, that a debugger and better error information is vital when a program crashes.

Another side note and I will stop. In the IBM Research compiler group, we needed a debugger for our compiler, and I was offered the opportunity write it. I declined.

I recall thinking, why would I want to do that? Instead, I would rather write a paper on using alpha-beta search to perform some specific code optimization.

I hope to influence a younger me in this audience that writing debuggers and tracing tools is worthwhile. And here, I will try to make it easier for you to write such debuggers and tools.
